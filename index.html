HTTP(S)扫盲帖-版本历史

Posted on 2019-01-07

HTTP 0.9

第一个版本的HTTP协议，极简只支持GET方法，不支持请求头限制了只能传输纯文本内容。

HTTP 0.9协议文档

HTTP 1.0

新增特性：

  • 请求与响应支持头域
  • 响应对象以一个响应状态行开始
  • 响应对象不只限于超文本
  • 开始支持客户端通过POST方法向Web服务器提交数据，支持GET、HEAD、POST方法
  • 支持长连接（但默认还是使用短连接），缓存机制，以及身份认证

特点缺点：

  • 无状态
    服务器不跟踪每个客户端也不记录过去的请求，可以借助cookie/session机制来做身份认证和状态记录。

  • 无连接
    每个事务独立处理，事务结束即释放连接（短连接），导致无法复用连接，TCP的建连和释放都耗时，网络的利用率偏低。

  • 队头阻塞（head of line blocking）
    1.0 版本要求下一个请求必须在前一个请求响应到达之前才能发送。

HTTP 1.1

新增特性（性能优化）：

  • 持久连接（Persistent Connection）
    增加Connection字段，避免每次客户端与服务器请求都要重复建立释放TCP连接，提高网络的利用率。

    • Connection: keep-alive, 开启HTTP持久连接，HTTP 1.1默认值

    • Connection: close, 关闭HTTP持久连接，HTTP 1.0默认值

  • 请求管道化（Pipelining）

    长连接使得请求管道化成为可能，实现“并行”请求。BUT，服务器必须按照客户端请求的先后顺序依次回送相应的结果，以保证客户端能够区分出每次请求的响应内容。

    这里的“并行”是把先进先出队列从客户端（请求队列）迁移到服务端（响应队列），但不存在两个并行的响应。

    浏览器厂商的大招！！！chrome的控制台可以看到并行请求。

    绿色部分代表请求发起到服务器响应的一个等待时间，而蓝色部分表示资源的下载时间。服务端对管道化的请求也必须进行逐个响应的传输数据，浏览器允许我们打开多个TCP的会话。上图中的并行，其实是不同的TCP连接上的HTTP请求和响应。

  • 增加缓存处理

    新增字段cache-control，实体标签e-tags等Cache的特性。

  • 增加Host字段

    虚拟主机的普及，是的一个物理服务器上存在多个共享一个IP地址的虚拟主机，所以需要引入Host头域来指定主机名（hostname）。

  • 支持断点传输

    • chunked编码传输
      该编码将实体分块传送并逐块标明长度，直到长度为0块表示传输结束, 这在实体长度未知时特别有用(比如由数据库动态产生的数据)。
    • 字节范围请求
      在请求消息中引入了range头域来实现，它允许只请求资源的某个部分。在响应消息中Content-Range头域声明了返回的这部分对象的偏移值和长度。如果服务器相应地返回了对象所请求范围的内容，则响应码206（Partial Content）。

  • 新增支持的方法

    OPTIONS，PUT，DELETE，TRACE，CONNECT

HTTP 2.0

HTTP 2.0最大的特点: 不会改动HTTP的语义，HTTP方法、状态码、URI及首部字段等等这些核心概念，致力于突破上一代标准的性能限制，改进传输性能，实现低延迟和高吞吐量。

  • 二进制分帧（2.0的由来）

    HTTP 2.0通过在应用层和传输层之间增加一个二进制分帧层，突破了HTTP 1.1的性能限制、改进传输性能。

    从上图来看，HTTP 2.0只是把原来传输的信息（HTTP 1.x的header和body）分割为更小的消息和帧（frame），并对它们采用二进制格式的编码。

  • 多路复用（连接共享）

    • 流（stream）：已建立连接上的双向字节流。
    • 消息：与逻辑消息对应的完整的一系列数据帧。

    • 帧（frame）：HTTP 2.0通信的最小单位，每个帧包含帧头部，至少也会标识出当前帧所属的流（stream id）。

      从图中可见，HTTP 2.0通信都在一个TCP连接上完成，这个连接可以承载任意数量的双向数据流。
      相应地，每个数据流以消息的形式发送，而消息由一或多个帧组成，这些帧可以乱序发送，然后再根据每个帧首部的流标识符（stream id）重新组装。

      多路复用（连接共享）可能会导致关键请求被阻塞。（优先权和依赖）

  • 头部压缩

    在HTTP 1.x中，头部元数据都是以纯文本的形式发送的，通常会给每个请求增加500~800字节的负荷。

    HTTP 2.0使用encoder来减少需要传输的header大小，通讯双方各自cache一份header fields表，既避免了重复header的传输，又减小了需要传输的大小。高效的压缩算法可以很大的压缩header，减少发送包的数量从而降低延迟。

  • 服务器推送（Server Push）

    客户端请求一个资源X，服务器端判断也许客户端还需要资源Z，无需客户端明确的请求下将资源Z推送到客户端，客户端接受到后，可以缓存起来以备后用。

  • 随时复位

    HTTP 1.1 一个缺点是当HTTP信息有一定长度大小数据传输时，你不能方便地随时停止它，中断TCP连接的代价是昂贵的。使用HTTP 2.0的RST_STREAM将能方便停止一个信息传输，启动新的信息，在不中断连接的情况下提高带宽利用效率。

  • 优先权和依赖

    HTTP 2.0里每个数据流都可以设置优先级和依赖，优先级高的数据流会被服务器优先处理和返回给客户端，数据流还可以依赖其他的子数据流。
    优先级别可以在运行时动态改变，当用户滚动页面时，可以告诉浏览器哪个图像是最重要的，你也可以在一组流中进行优先筛选，能够快速抓住重点流。

创建Git图床

Posted on 2018-09-03

利用 github 的 wiki 来管理独立图床，而非通过创建单独的图床repo或者在当前工程repo中将图片和代码放在一起。

  1. 创建 Wiki 页面

    在创建图床的项目下，点击项目的 wiki 标签

    确保已经创建了 wiki 页面

  2. Clone Wiki

    从Clone this wiki locally拿到git url

    $ git clone https://github.com/[username]/projectname.wiki.git

  3. 存放图片到库一级目录或创建二级图片目录

    建议创建一个图库的单独目录 images

  4. 上传Git管理

    常见的 git 提交操作

  5. 引用图片

    完整路径访问：

    https://raw.githubusercontent.com/wiki/[username]/[repository]/[folderifexsist]/[filename]

    wiki文档访问：

    [folderifexsist]/[filename]

同一设备管理多个Git帐号

Posted on 2018-09-03

前言

在日常开发中，我们一定会遇到使用多个Git帐号或者平台的情况。比如自己造轮子的github，干私活能有免费私有仓库gitee，工作中公司搭建的私有git平台等等。那么希望在一台设备上提交不同平台的代码，我们就一定要知道如何配置SSH Key来实现验证流程（其实也不一定，使用一些成熟的可视化代码管理工具如 source tree也是可以的）。

配置SSH Key(github为🌰)

打开你的终端，到 .ssh 目录

$ cd ~/.ssh

常规方法生成一个 SSH Key

$ ssh-keygen -t rsa -C “your@email.com“

自定义生成key的名字

名字定义要清晰，因为你的设备下可能会有多个rsa文件。

密码的设置

看自己喜好

配置github或者其他平台

复制公钥

$ pbcopy < ~/.ssh/test_rsa.pub

到github或者其他平台的设置项中，找到SSH and GPG keys 选项执行 New SSH key的操作。

测试

键入 ssh -T git@github.com，一般能够看到类似下面的输出。

$ ssh -T git@github.com

Hi [UserName]! You’ve successfully authenticated, but GitHub does not provide shell access.

如果你非常不幸，没能看到这样的输出，你可以直接跳到下一环节。或者执行下面一段命令将新ssh-key添加到ssh agent中，再执行测试命令。

$ ssh-add ~/.ssh/test_rsa
